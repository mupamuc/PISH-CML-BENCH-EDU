# Правила проекта

Проект — часть общей работы над сильной системой управления университетом (ИСУ).
Два репозитория, одна система:

- **mupamuc/PISH-CML-BENCH-EDU** (этот репо) — доменная модель и демо-стенд магистратуры ПИШ «Системный цифровой инжиниринг БАС»: учебный план как данные, сквозной проект, V&V-приёмка, компетенции, глоссарий, грант-радар.
- **LeVa-cloud/university-saas** — производственная платформа: WordPress-плагин `university-lk`, роли/права, instance-per-tenant (152-ФЗ), деплой на Jino.

Главный документ: [PLAN-university-saas-integration.md в LeVa-cloud/university-saas](https://github.com/LeVa-cloud/university-saas/blob/main/docs/PLAN-university-saas-integration.md) — детальный план переноса идей стенда в university-saas. Канонический экземпляр живёт в репозитории реализации (`university-saas`); локальный [docs/PLAN-university-saas-integration.md](docs/PLAN-university-saas-integration.md) — только указатель. Любая работа сверяется с планом; изменения плана — отдельным коммитом с обоснованием.

## Правила разработки (обязательные)

1. **Вся разработка — через ветки.** Никаких коммитов напрямую в `main`.
   Ветки: `feature/<короткое-имя>` для фич, `fix/<имя>` для исправлений.
   В university-saas ветки создаются от `dev` (по его WORKFLOW.md), здесь — от `main`.
2. **Мердж с `main` — потом**, после проверки: работа завершена, проверена локально (для university-saas — на docker-тенанте), описана в PR/коммите. В `main` попадает только рабочее состояние, готовое к показу/деплою.
3. **Разбиваем на отдельные проекты (модули)** и разрабатываем общую сильную систему.
   Каждый модуль — самостоятельный кусок с чёткой границей (curriculum, projects, review, analytics, glossary, grants, ideas — см. фазы плана). Один модуль = одна ветка = один осмысленный мердж. Не смешиваем модули в одной ветке.
4. **Фронтенд — в конце.** Сначала модель данных, права, серверная логика и отчёты; полировка UI — фаза 7 плана.
5. **Конвенции university-saas соблюдаем строго:** capabilities через версионирование ролей, формы через `admin-post.php` + nonce + серверная проверка прав, тяжёлые операции — порционный WP-Cron, без Composer и JS-фреймворков, удаление — только administrator.
6. **Коммиты** — conventional commits (`feat:`, `fix:`, `docs:`, `refactor:`…), тема на русском, по одному логическому изменению на коммит.
7. **Секреты в git не попадают** (пароли БД, ключи, реальные `.env`). Данные студентов — только в MySQL тенанта, никаких внешних сервисов без решения по 152-ФЗ.

## Распределение

Дальнейшую реализацию плана в university-saas ведёт Левенцов (LeVa-cloud). Этот репо остаётся источником доменной модели, датасетов (db.js → фикстуры) и публичным демо-контуром.
