# План интеграции идей PISH-CML-BENCH-EDU в university-saas

**Дата:** 2026-07-18
**Репозитории:**
- Источник идей: [mupamuc/PISH-CML-BENCH-EDU](https://github.com/mupamuc/PISH-CML-BENCH-EDU) — статический демо-стенд магистратуры ПИШ «Системный цифровой инжиниринг БАС» (ванильный JS + Supabase для доски идей).
- Целевая платформа: [LeVa-cloud/university-saas](https://github.com/LeVa-cloud/university-saas) — ИСУ (SaaS для университетов), WordPress-плагин `university-lk`, instance-per-tenant, прод на shared-хостинге Jino.

**Договорённость:** фронтенд — в самом конце, сначала устаканиваем содержание (модель данных, модули, права, отчёты).

## 1. Стратегическая рамка

### 1.1. Почему проекты дополняют друг друга

| | PISH-CML-BENCH-EDU | university-saas |
|---|---|---|
| Сильная сторона | Доменная модель обучения: учебный план как данные, сквозной проект, артефакты, V&V, компетенции, детектор дублей, глоссарий, грант-радар | Производственный каркас: роли/права, мультитенантность, импорт xlsx, WP-Cron, деплой, журналирование, 152-ФЗ |
| Слабая сторона | Всё демо: нет авторизации, нет персистентности (кроме доски идей), данные захардкожены | Учебное ядро пустое: CPT `course`/`schedule_entry` — заготовки; кабинеты студента/преподавателя почти пустые |
| Хранение | JS-объекты в браузере + Supabase (3 таблицы) | WP CPT + post meta + 1 своя таблица (MySQL) |

**Тезис плана:** перенести из PISH-стенда не код, а **модель данных и механики**, реализовав их средствами WordPress-плагина в существующих конвенциях коллеги (CPT + meta, `admin-post.php` + nonce, capability-модель через `university_lk_active_role_can()`, версионирование ролей, WP-Cron для тяжёлых операций).

### 1.2. Что переносим (карта идей)

| # | Идея из PISH-стенда | Куда ложится в university-saas | Ценность | Сложность |
|---|---|---|---|---|
| И1 | Учебный план как данные: дисциплины, модули, семестры, темы (лекции/практики с часами), РПД | Наполнение пустого CPT `course` → новые CPT `discipline`, `discipline_topic`; импорт из xlsx (механизм импорта уже есть в модуле «Нагрузка») | Высокая: закрывает главный TODO коллеги («спроектировать сущности курсов») | Средняя |
| И2 | Компетенции ФГОС (УК/ОПК/ПК) + связи дисциплина↔компетенция + отчёт покрытия | CPT `competency` + связи в meta; отчёт для РОП/ректората (cap `view_reports`) | Высокая: уникальная аналитика, готовый алгоритм | Низкая-средняя |
| И3 | Карта владения темами + детектор дублей (тема излагается вне дисциплины-владельца) | Meta «владелец темы» + отчёт-детектор с ранжированием по score | Высокая: киллер-фича для РОП, ни в одной LMS нет | Средняя |

| И6 | Глоссарий с источниками (ГОСТ/ПИШ/курс) и дисциплиной-владельцем + сквозные тултипы | CPT `glossary_term`; тултипы — в фазе фронтенда | Средняя | Низкая |
| И7 | Грант-радар: каталог грантов, матчинг по ключевым словам и проектам, service-desk заявок | Расширение существующего модуля «Конкурсы студентов» (CPT `competition` уже есть, статусы заявок уже заложены) | Средняя-высокая | Низкая-средняя |
| И8 | Банк идей: голосование, комментарии, разделы | Расширение существующего модуля «Предложения» (CPT `suggestion`) | Средняя | Низкая |
| И9 | Трекер когорты: пайплайн стадий, риски команд, тепловая карта | Отчётный экран поверх И4 (данные те же) | Средняя | Низкая после И4 |
| И10 | Читалка материалов дисциплины + тест самоконтроля («надстройка над Moodle») | Отдельный модуль контента; поздняя фаза | Средняя | Высокая |
| И11 | ИИ-тьютор с «честным контуром» | Отложено: требует внешнего API, на Jino рискованно; оставляем как расширение | Потенциально высокая | Высокая |
| И12 | Радар компетенций студента, силовой граф идей, живой тикер | Фронтенд-фаза (в конце) | Косметика/вау | Средняя |

### 1.3. Что сознательно НЕ переносим (и почему — честно)

- **Supabase и realtime.** В university-saas данные обязаны жить в MySQL тенанта (152-ФЗ, физическая изоляция). Realtime-графы на shared-хостинге без websockets не сделать; polling — можно, но это фронтенд-вопрос последней фазы.
- **Архитектуру «всё в браузере».** Демо-подход стенда (данные в `window.*`) не переносится — всё становится CPT/meta.
- **Пайплайн `build_db.py` как есть.** Идею (автогенерация БД из `.plx` и РПД) переносим, но реализуем как xlsx/CSV-импорт в стиле уже существующего парсера коллеги (`ZipArchive` + `SimpleXML`, без Composer). Парсить `.plx` напрямую на PHP — можно позже; на старте достаточно промежуточного xlsx.
- **CRM-чат.** У чата в WP на shared-хостинге плохое соотношение цена/ценность (нет push, только polling). Вместо него — комментарии к сущностям (заявкам, артефактам). Если чат реально нужен — отдельное решение, не в этом плане.

### 1.4. Слабые места, которые надо обсудить с коллегой до старта

1. **Один плагин уже большой.** `university-lk` растёт монолитно. Предлагаю внутри плагина ввести подпапки `includes/modules/<module>/` и реестр модулей, иначе через 3 фазы получим 30 файлов в одной куче. Это рефакторинг структуры, не логики — дёшево сейчас, дорого потом.
2. **CPT + meta против собственных таблиц.** Для тем дисциплин (сотни строк × десятки тенантов) и связей дисциплина↔компетенция (сотни связей) meta-запросы становятся медленными и корявыми (JOIN по postmeta). Прецедент своей таблицы уже есть (`university_lk_pps_log`). Рекомендация: **связи и темы — в собственные таблицы через `dbDelta()`**, сущности первого класса (дисциплина, команда, артефакт, конкурс) — CPT. Это решение надо принять в Фазе 0, менять потом больно.
3. **Терминология.** У коллеги «course» — заготовка; у нас «дисциплина». Предлагаю: CPT `discipline` (дисциплина учебного плана), а `course` либо удалить, либо оставить под «поток/реализацию дисциплины в конкретном году». Зафиксировать словарь в docs.
4. **Мульти-программность.** PISH-стенд — про одну программу. ИСУ — про вуз, где программ много. Все новые сущности сразу проектируем с полем «программа» (taxonomy `academic_program`), даже если в первом тенанте программа одна.

## 2. Целевая модель данных (итог всех фаз)

### 2.1. Новые CPT

| CPT | Заголовок | Ключевые meta (`_disc_*`, `_team_*`, `_art_*`, ...) | Taxonomy |
|---|---|---|---|
| `discipline` | Название дисциплины | код (`К.М.01.03`), модуль, семестр(ы), з.е., часы, форма контроля, роль в проекте, вклад в проект, границы («что НЕ входит»), `has_rpd`, `is_core`, кафедра | `academic_program`, `curriculum_module` |
| `project_team` | Название команды | программа, год набора, тип изделия, участники (user_id + роль в команде), куратор (user_id), текущая стадия (S1–S4), статус риска (ok/warn/risk) + причина, тема НИР/ВКР (+флаг «не зафиксирована») | `academic_program` |

### 2.2. Собственные таблицы (по прецеденту `pps_log`)

- `{prefix}university_lk_topics` — темы дисциплин: id, discipline_id, тип (лекция/практика/лаба), номер, название, часы, семестр, владелец-флаг. (~200–500 строк на программу.)

## 3. Фазы работ

> Оценки — в рабочих днях одного разработчика, грубо. Каждая фаза заканчивается: (а) деплоем на dev-тенант, (б) прогоном чек-листа приёмки фазы, (в) merge `dev`→`main` по WORKFLOW.md коллеги.

### Фаза 0 — Фундамент (2–3 дня)

Цель: договориться об архитектуре, чтобы дальше не переделывать.

1. **Решения (совместно с коллегой, фиксируем в `docs/ADR/`):**
   - ADR-001: CPT vs собственные таблицы (предложение — гибрид, см. §1.4 п.2).
   - ADR-002: судьба CPT `course`/`schedule_entry` (переименовать/удалить/оставить под расписание).
   - ADR-003: словарь терминов (дисциплина, модуль, артефакт, стадия, команда, программа).
   - ADR-004: структура плагина — `includes/modules/<name>/{module.php, admin.php, render.php}` + реестр модулей; миграция существующих файлов НЕ делается (только новые модули по новой структуре), чтобы не ломать прод.
2. **Инфраструктура:**
   - Ветка `feature/curriculum` от `dev`.
   - Поднять docker-tenant локально, завести тестовые учётки всех ролей.
   - Заготовка механизма собственных таблиц: общий хелпер версионирования схемы (обобщить подход `pps_log_db_version`).
   - Taxonomy `academic_program` + страница управления программами (минимальная, в wp-admin — это инструмент админа).
3. **Данные для теста:** выгрузить из PISH-стенда `db.js` → промежуточные CSV/xlsx (дисциплины, темы, компетенции, связи) — это готовый реалистичный датасет (44 дисциплины, 224 темы, 15 компетенций, 185 связей).

**Критерий готовности:** ADR приняты, docker-тенант у обоих запускается, тестовый датасет в xlsx лежит в репо (`docs/fixtures/`).

### Фаза 1 — Учебный план: дисциплины, темы, компетенции (И1, И2 частично) (5–7 дней)

Модуль `curriculum`. Наполняет пустое учебное ядро ИСУ.

1. CPT `discipline` + taxonomy `curriculum_module` + все meta (§2.1). CRUD-экраны в кабинете по образцу модуля «ППС» (карточка + таблица с группировкой по модулю/семестру).
2. Таблицы `topics` и `disc_comp` + CRUD тем внутри карточки дисциплины.
3. CPT `competency` + привязка компетенций к дисциплине (мультиселект в карточке).
4. **Импорт xlsx** (переиспользовать конвейер модуля «Нагрузка»: сопоставление колонок по ключевым словам, лимиты, порционный WP-Cron, экран прогресса):
   - лист «Дисциплины» → CPT;
   - лист «Темы» → таблица topics;
   - лист «Компетенции» и «Связи» → competency + disc_comp.
   - Импорт — только `administrator` (как у нагрузки).
5. Витрина учебного плана для всех ролей (`view_curriculum`): таблица дисциплин с фильтрами по модулю/семестру (повторить UX фильтров из модуля «Нагрузка» — уже написанный JS-паттерн), карточка дисциплины: программа тем, часы, компетенции, роль в проекте, границы.
6. Связка с модулем «Нагрузка»: у `workload_entry` есть дисциплина строкой — добавить мягкое сопоставление с CPT `discipline` (по названию, как сопоставляются ФИО в ППС), meta `_wl_discipline_id`. Не блокирующее: несопоставленные строки живут как раньше.

**Критерий готовности:** датасет ПИШ (44/224/15/185) импортируется без ручных правок; студент видит план своего семестра; дирекция редактирует дисциплину; связка нагрузка↔дисциплина работает на реальной выгрузке коллеги.

**Риски:** качество исходных xlsx (лечится — конвейер коллеги уже переживал «Плутон» на 1800 строк); объём meta-полей дисциплины (держать реестром полей, как `university_lk_workload_fields()`).

### Фаза 2 — Сквозной проект: команды, стадии, артефакты (И4) (6–8 дней)

Модуль `projects`. Главная концептуальная новация.

1. CPT `project_team`: карточка команды (состав из пользователей WP с ролями «главный конструктор / расчётчик / автономность / технолог / системный инженер» — список ролей редактируемый, в опции программы), куратор, тип изделия, стадия, риск + причина, тема НИР с флагом «не зафиксирована».
2. Справочник стадий per-программа (опция): код, название, семестр, описание. Дефолт — С1–С4 из ПИШ.
3. CPT `project_artifact`: привязка к команде + дисциплине + семестру + стадии; ответственный студент; файлы через `media_handle_upload` (паттерн из «Конкурсов»); статусная модель `draft → submitted → in_review → accepted/rejected` (возврат `rejected → draft` с комментарием).
4. **Автогенерация плана артефактов:** при создании команды — кнопка «Сформировать план артефактов из учебного плана» (по meta «вклад в проект» дисциплин семестра создаются draft-артефакты). Это связка Фазы 1 и 2, изюминка переноса.
5. Кабинет студента: «Моя команда» (состав, стадия, тема НИР), «Мои артефакты» (список, загрузка файлов, отправка на приёмку).
6. Кабинет куратора/РОП: список команд, смена стадии (с проверкой: нельзя на следующую стадию при непринятых обязательных артефактах текущей — мягкое предупреждение, не жёсткий блок), выставление риска.
7. Журнал `artifact_log` со всеми событиями (по образцу журнала ППС, с группировкой по дням).

**Критерий готовности:** сквозной сценарий на dev-тенанте: дирекция создаёт команду → генерирует план артефактов → студент загружает файл и отправляет → артефакт виден преподавателю в очереди приёмки (сама приёмка — Фаза 3, пока просто accept/reject кнопкой).

**Риски:** членство в команде требует, чтобы студенты были заведены как пользователи — нужен массовый импорт студентов (мини-задача: CSV-импорт пользователей с ролью student, по образцу «Приглашения ППС»).

### Фаза 3 — V&V-приёмка и workflow согласований (И5) (4–5 дней)

Модуль `review`. Использует давно объявленный, но нереализованный cap `approve_workflow`.

1. Справочник шаблонов чек-листов V&V (per-программа, редактирует дирекция/РОП): пункты цепочки достоверности (постановка задачи, сходимость сетки, валидационный базис, оценка неопределённости, ссылка на ГОСТ Р 57700.xx…). Дефолтный шаблон — из `app.html` стенда (vvList).
2. Привязка шаблона чек-листа к типу артефакта/дисциплине.
3. Экран приёмки у преподавателя (`review_artifacts`): очередь `submitted`-артефактов по своим дисциплинам (через связку нагрузка↔дисциплина из Фазы 1 — преподаватель видит артефакты дисциплин, где он закреплён в ППС!), карточка приёмки: файлы, чек-лист с чекбоксами, **кнопка «Принять» активна только при полностью закрытом чек-листе** (проверка и в UI, и на сервере — как окно ППС), «Вернуть» — с обязательным комментарием.
4. Каждый чекнутый пункт и вердикт — в `artifact_log` (кто, когда) — это и есть «цепочка достоверности» с аудитом.
5. Уведомления `wp_mail()`: студенту при accept/reject (паттерн из «Предложений»).

**Критерий готовности:** невозможно принять артефакт с незакрытым чек-листом ни через UI, ни прямым POST; журнал показывает полную историю; преподавателю попадают только его дисциплины.

**Отмечу отдельно:** связка «закрепление ППС → очередь приёмки преподавателя» — это то, чего нет ни в стенде, ни в ИСУ по отдельности; она возникает именно из объединения проектов.

### Фаза 4 — Аналитика программы: покрытие, дубли, трекер когорты (И2, И3, И9) (5–6 дней)

Модуль `analytics` (cap `view_program_analytics`).

1. **Покрытие компетенций:** таблица «компетенция × сколько дисциплин формирует», подсветка нулей (пробелы), фильтр по программе/семестру. Алгоритм тривиален поверх `disc_comp`. Экспорт в CSV.
2. **Детектор дублей:** портировать логику score из стенда (совпадение ключевых слов темы с темами чужой дисциплины-владельца) в PHP; пересчёт по кнопке + еженедельный WP-Cron; результат — в `topic_dups`; экран разбора для РОП: подтвердить дубль / отклонить с комментарием (статус хранится, повторный пересчёт не воскрешает отклонённые).
3. **Трекер когорты:** пайплайн-полоса стадий (сколько команд на каждой), таблица команд с прогрессом (принятые/всего артефактов), риски с причинами, сигнальная сводка «незафиксированные темы НИР». Публичной витрины (как tracker.html) НЕ делаем — ИСУ закрытая; но закладываем возможность анонимизированного экспорта для лендинга.
4. Дашборд ректората (`view_reports` наконец обретает смысл): агрегаты по программам — покрытие, стадии, % принятых артефактов, отклонения нагрузки (уже есть).

**Критерий готовности:** на датасете ПИШ детектор находит те же ~16 кандидатов-дублей, что и стенд; отчёт покрытия совпадает с эталоном из `db.js`; трекер отражает тестовые команды Фазы 2.

### Фаза 5 — Глоссарий, банк идей, грант-радар (И6, И7, И8) (4–6 дней)

Три независимых улучшения существующих модулей — можно параллелить или резать по приоритету.

1. **Глоссарий:** CPT `glossary_term`, CRUD (`manage_glossary`), импорт из `glossary.js` стенда (готовые ~25 терминов ГОСТ/ПИШ — сразу полезный контент), страница глоссария в кабинете с поиском и фильтром по категории/источнику. Поле «синонимы» заполняем сразу — пригодится тултипам в фронтенд-фазе.
2. **Предложения → Банк идей:** к CPT `suggestion` добавить: раздел (taxonomy), тип (идея/критика), голоса (таблица или meta-счётчик + user meta против повторов — перенос механики PK(idea, client) из Supabase на user_id), комментарии (WP comments у CPT). Сортировка «новые/популярные». Существующий admin-workflow «выполнено + письмо» сохраняется без изменений.
3. **Конкурсы → Грант-радар:** к CPT `competition` добавить meta «ключевые слова»; экран «Радар» для РОП: матчинг конкурсов на команды (ключевые слова × тип изделия/тема НИР из Фазы 2), кнопка «Направить команде» создаёт `competition_entry` со статусом submitted и письмом куратору. Существующий таймлайн по месяцам уже есть у коллеги — переиспользуем. Статусная цепочка заявки (`submitted→supported→winner/rejected`) уже заложена — задействуем полностью, это и есть service-desk из стенда, только на WP-механике.

**Критерий готовности:** глоссарий ПИШ импортирован; идея получает голос ровно один раз на пользователя; радар предлагает релевантные конкурсы тестовой команде.

### Фаза 6 — Контент дисциплин: материалы и тесты самоконтроля (И10) (7–10 дней, можно отложить)

Модуль `content`. Самая тяжёлая и наименее срочная содержательная фаза — включать после того, как Фазы 1–5 обкатаны на реальном тенанте.

1. Материалы темы: файлы + встроенные страницы (WP-редактор) с привязкой к теме/дисциплине; видимость управляется преподавателем (паттерн тумблеров из стенда).
2. Тесты самоконтроля: вопрос/варианты/пояснение (портировать структуру `LECTURE_CFD.quiz`), прохождение без оценивания, моментальное пояснение; результаты — только студенту (принцип «тренировка, не контроль» из стенда).
3. Решить с коллегой: глубина против Moodle. Позиция стенда («надстройка, не замена») здесь верна: НЕ строить SCORM/оценивание, только чтение + самоконтроль. Если у вуза Moodle — ссылки/LTI позже.

### Фаза 7 — Фронтенд (И12 + полировка всего) (по отдельному плану)

По договорённости — в конце. Сейчас только фиксируем кандидатов:
- Тултипы терминов глоссария по всем страницам кабинета (движок `tooltips.js` стенда портируется почти как есть — он ванильный JS, данные возьмёт из CPT через локализацию скрипта).
- Радар компетенций студента (SVG, из стенда) — потребует данных «подтверждено артефактами» (появятся после Фазы 3: принятые артефакты → компетенции дисциплины).
- Тепловая карта команды×стадии, пайплайн-полоса — статические SVG/CSS, без библиотек (в духе no-framework подхода коллеги).
- Дизайн-система: у коллеги DESIGN.md v3 «деловой отчётный» — новые модули верстаем в нём, ничего из тёмной темы стенда не тянем без обсуждения.
- Силовой граф идей — только если останутся силы; на WP он живёт как отдельная страница с polling.

**ИИ-тьютор (И11) — вне плана**, отдельное решение после Фазы 6: нужен внешний API, бюджет, и политика «честного контура»; на shared-хостинге прокси к LLM возможен, но это самостоятельный проект.

## 4. Сводный график и порядок

```
Фаза 0 (2–3д) → Фаза 1 (5–7д) → Фаза 2 (6–8д) → Фаза 3 (4–5д) → Фаза 4 (5–6д) → Фаза 5 (4–6д) → [Фаза 6 (7–10д)] → Фаза 7 (фронтенд)
```

Итого ядро (Фазы 0–5): **26–35 рабочих дней** одного разработчика; вдвоём с разделением (один — curriculum/analytics, второй — projects/review) — реалистично 3,5–5 недель календарно. Фазы 4 и 5 частично параллелятся с 3.

Первый осязаемый результат для показа заказчику — конец Фазы 1 (живой учебный план в ИСУ), первый «вау» — конец Фазы 3 (V&V-приёмка с блокировкой).

## 5. Правила работы (по конвенциям коллеги)

- Ветки `feature/*` от `dev`; в `main` — только после проверки на docker-тенанте; деплой на Jino — только из `main` через `provision-tenant.sh`/rsync (WORKFLOW.md).
- Новые модули — по структуре ADR-004; существующие файлы не трогаем без необходимости.
- Все новые capability — через поднятие `UNIVERSITY_LK_ROLES_VERSION` (механизм уже есть).
- Все тяжёлые операции (импорты, пересчёт дублей) — порционный WP-Cron (лимит 30 с на Jino).
- Все формы — `admin-post.php` + nonce + серверная проверка `university_lk_active_role_can()`; удаление — только administrator.
- Никаких Composer-зависимостей и JS-фреймворков (ограничения Jino и стиль проекта).
- Фикстуры для тестов — датасет ПИШ из `db.js` (экспорт в xlsx, Фаза 0).
